# 0003 — `Ref<T>` primitive の実装方針

## Status

Accepted — 2026-04-23

## Context

JSX で描画した DOM 要素への参照を外から受け取る仕組みが必要。典型的なユースケース:

- `onMount(() => input.focus())` 時に input 要素を掴みたい
- 要素のサイズ計測 (`getBoundingClientRect()`)
- 外部ライブラリ (Chart.js 等) に DOM 要素を渡したい

Vidro には現状、DOM 要素を外から掴む手段がなく、`const el = (<input />) as HTMLInputElement`
で JSX 式を変数に受ける不自然な書き方に頼るしかない。

論点が 3 つある:

1. **API 形式** — 新 primitive (箱) を追加するか、コンパイラ糖衣で変数代入に変換するか
2. **アクセス API** — `.current` (React 流) / `.value` (Signal と同じ)
3. **factory / callback 形式** — class のみ / factory も / 関数 callback も

## Options

### 論点 1: API 形式

- **A. Solid 方式** (コンパイラ糖衣): `let input!: HTMLInputElement; <input ref={input} />`
  - Babel plugin で `ref={input}` → `(el) => input = el` に変換
- **B. 関数 callback のみ**: `let input; <input ref={(el) => input = el} />`
  - Runtime のみで完結、コンパイラ不要
- **C. new Ref 方式** (箱を作る): `const input = new Ref<HTMLInputElement>(); <input ref={input} />`
  - Signal と同じ class + factory パターン

### 論点 2: アクセス API

- **A. `.current`** (React 流)
- **B. `.value`** (Signal と同じ)

### 論点 3: サポートする作成形式

- **A. class のみ** (`new Ref()`)
- **B. class + factory** (`new Ref()` / `ref()`)
- **C. class + factory + 関数 callback** (`<input ref={(el) => ...} />` も可)

## Decision

- 論点 1 → **C (new Ref 方式)**
- 論点 2 → **A (`.current`)**
- 論点 3 → **B (class + factory)**

## Rationale

### 論点 1: new Ref 方式

3 案の比較:

|                        | A (Solid)           | B (callback)     | C (new Ref)            |
| ---------------------- | ------------------- | ---------------- | ---------------------- |
| 概念の追加             | しない              | しない           | 1 個増える             |
| 型安全性               | `!` で TS を騙す    | `!` で TS を騙す | 型が正直 (`T \| null`) |
| 宣言の明示性           | let 変数 (意図不明) | let 変数         | `new Ref()` (明示)     |
| Signal との API 一貫性 | 別                  | 別               | 同じ (class + factory) |
| コンパイラ作業         | 必要                | 不要             | 不要                   |

- A は「引き算のデザイン」哲学に沿うが、**TypeScript に non-null assertion (`!`) で嘘を
  つかせる**のが `型貫通` 哲学と矛盾。mount 前に `input.focus()` を呼ぶと ランタイム
  crash するが TS は catch できない
- B は A の下位互換 (コンパイラなしで動く版)。書くのが冗長
- C は新 primitive を 1 個増やすが、**型安全 (`T | null` で null チェックを強制)** かつ
  **宣言で意図が明示される** (`new Ref()` を見れば ref だと一目瞭然)。Signal と同じ
  class + factory パターンで **"Vidro の primitive は `new Xxx` / `xxx()` で作る"** の
  統一感も出る
- **判断軸**: Vidro の 5 哲学のうち `型貫通` と `AI-native 規約` (= 明示性・可読性) を
  `引き算のデザイン` より優先。概念 1 つの追加は許容範囲

### 論点 2: `.current`

- B (`.value`) は Signal と同じ API になる → **Signal と Ref を見分けるコストが発生**
  (型を見ないと区別できない)
- A (`.current`) は React と同じ。**見た瞬間に「ref」と分かる**。実運用の可読性で勝つ
- **判断軸**: 区別の明示性 > 形式の統一

### 論点 3: class + factory

- Signal が既に class + factory 両対応 (`project_signal_api_decision` memory 参照)
- Ref も同じパターンで統一。学習コスト削減
- 関数 callback 形式 (`ref={(el) => ...}`) は YAGNI で今回は見送り。必要になったら
  applyProp に分岐を足すだけで追加できる
- **判断軸**: 既存パターンへの準拠、YAGNI

## Consequences

- `Ref<T>` という新しい primitive が公開 API に追加される (概念 +1)
- `applyProp` に `key === "ref"` の早期 return 分岐が増える
- DOM 要素が Show の detach / dispose で外されても `.current` は null に**戻らない**
  (代入した参照はそのまま残る)。これは YAGNI で簡略化。unmount 検知が必要なら
  onCleanup と組み合わせる (ユーザー側で明示)
- コンポーネント (`<MyComponent ref={r} />`) への ref forwarding は未対応。intrinsic
  element (`<input ref={r} />`) のみサポート

## Revisit when

- **関数 callback 形式が必要になった時**: `<input ref={(el) => ...} />` を使いたい
  ユースケースが出てきたら applyProp に `typeof value === "function"` の分岐を追加
- **コンポーネントへの ref forwarding が必要になった時**: props の flow を変える
  必要があるため、forwardRef 相当の機構を別 ADR で検討
- **unmount 時に `.current = null` に戻したい要件が出た時**: applyProp から
  onCleanup を登録する形に拡張
