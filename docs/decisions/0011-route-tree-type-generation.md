# 0011 — Route tree 型生成 plugin + props 完結方針

## Status

Accepted — 2026-04-24

## Context

設計書 3.6「Route params: directory から自動生成された型」に到達するため、
`routes/` ディレクトリ構造から URL pattern と params 型を抽出する仕組みを入れる。
前 session で `@vidro/plugin` を切り出したので、その自然な拡張として
`routeTypes()` plugin を追加する流れ。

これを入れると:

- `/users/:id` にアクセスする loader で `params.id` が `string` として補完される
- ファイルパス `[id]` のタイポや名前変更が **型エラーとして落ちる** (route が存在
  しない key 参照になる)
- 将来 `useParams` 系 API を足す場合にも同じ `Routes` 型が土台になる

論点は 4 つ:

1. **出力方式**: 生成された型をどこに吐くか (virtual module vs 物理 .d.ts)
2. **型の置き場**: `Routes` を新 export するか、既存 `@vidro/router` に
   module augmentation するか
3. **param にアクセスする API**: hook (`useParams()`) を作るか、component の
   props で受け取って完結させるか
4. **loader 引数で route path をどう指定するか**: `LoaderArgs<R>` helper を
   残すか、辞書を直接書かせるか、`defineLoader` helper で runtime に逃がすか

## Options

### 論点 1: 出力方式

- **1-A.** virtual module: `import { Routes } from "virtual:vidro/routes"`
  - git に残らない、ファイル増えない
  - vite dev server / vite plugin が resolve しないと型が出ない
    (tsserver 単体で閉じないため IDE 起動状態依存)
- **1-B.** 物理生成: `src/.vidro/routes.d.ts` を plugin が書き出す
  - tsserver が単独で解決できる (IDE 再起動だけで補完が戻る)
  - watch + 再生成が必要、gitignore も必要
  - ファイルが 1 枚増える

### 論点 2: 型の置き場

- **2-A.** 新規 type export: `@vidro/router` が `Routes` を export
  - 明示的、追跡しやすい
  - 生成ファイル側で `@vidro/router` を import してから上書き augment する形になり、
    依存方向が循環気味
- **2-B.** module augmentation: 生成ファイルが `@vidro/router` の `RouteMap`
  interface を `declare module` で augment
  - 既存 export の拡張という構造が TS 的に自然 (`csstype`, `vite/client` 等と同じ)
  - 生成ファイルは単なる declaration なので runtime 成果物ゼロ

### 論点 3: param アクセスの API

- **3-A.** hook `useParams<R extends keyof Routes>()` を @vidro/router で提供
  - React Router 流、慣れた people にとってはおなじみ
  - **Vidro は invoke-once** で use 接頭辞も他に無く、文化的に浮く
  - 深い子孫 component で呼ばれた際、**call site の route が型レベルで決まらない**
    ので、leaf component 内 hook に限定しないと型が緩くなる
  - hook で型安全を達成するには plugin codegen で route 情報をファイル単位で注入
    する必要がある (TanStack Router 流、魔法度が高い)
- **3-B.** props 完結: Router が leaf / layout に `params` を props で渡す現状
  のまま、hook は提供しない
  - 「component は props で受ける」の一本化、Vidro の透明性哲学に合う
  - 深い子孫で param が欲しければ **props でバケツリレー** または **signal で足す**
    (YAGNI で後回し可)
  - 型は `PageProps<typeof loader>` 経由で loader 引数から自動推論される構造が
    既にあり、そこに Routes を繋ぐだけで完結する

### 論点 4: loader 引数の書き方

- **4-A.** 既存 helper を拡張: `LoaderArgs<R extends keyof Routes>`

  ```ts
  export async function loader({ params }: LoaderArgs<"/users/:id">) {}
  ```

  - 短い、既存 `LoaderArgs` 型を壊さない (generic を絞るだけ)
  - 将来 `request` / `context` が loader 引数に増えたとき 1 箇所変更で全 route 追従

- **4-B.** 辞書を直接書く: `{ params: Routes["/users/:id"]["params"] }`

  ```ts
  export async function loader({ params }: { params: Routes["/users/:id"]["params"] }) {}
  ```

  - helper 1 枚を剥がして「Routes 辞書を引いてるだけ」が可視化される
  - `params` を 2 回書くのが冗長、loader 引数拡張 (request 追加等) に全 route 追従
    の仕組みが無い

- **4-C.** `defineLoader` helper (runtime 関数で route path を受ける)

  ```ts
  export const loader = defineLoader<"/users/:id">(async ({ params }) => { ... })
  ```

  - 関数呼び出しの generic として path を渡す、hover 体験は自然
  - runtime に wrapper 関数が乗る (identity call でも import / export 形が変わる)
  - `export async function loader` の素朴さを失う

## Decision

- 論点 1 → **1-B (物理生成、`src/.vidro/routes.d.ts`)**
- 論点 2 → **2-B (module augmentation、`@vidro/router` に `RouteMap` interface)**
- 論点 3 → **3-B (props 完結、hook は作らない)**
- 論点 4 → **4-A (`LoaderArgs<R extends keyof Routes>` で helper 拡張)**

### 公開 API

```ts
// @vidro/router が export する interface (初期状態は空)
export interface RouteMap {}

// 生成ファイルが augment する
export type Routes = RouteMap;

// LoaderArgs は Routes の key を受け取る形に拡張
export type LoaderArgs<R extends keyof Routes = keyof Routes> = {
  params: Routes[R]["params"];
};
```

### 生成される `src/.vidro/routes.d.ts`

```ts
// AUTO-GENERATED by @vidro/plugin routeTypes() — DO NOT EDIT
declare module "@vidro/router" {
  interface RouteMap {
    "/": { params: {} };
    "/about": { params: {} };
    "/users": { params: {} };
    "/users/:id": { params: { id: string } };
  }
}
export {};
```

### ユーザー側の書き方

```ts
// routes/users/[id]/server.ts
import type { LoaderArgs } from "@vidro/router";

export async function loader({ params }: LoaderArgs<"/users/:id">) {
  // params: { id: string } ← Routes 辞書から展開
  return await getUser(params.id);
}
```

```tsx
// routes/users/[id]/index.tsx
import type { PageProps } from "@vidro/router";
import type { loader } from "./server";

export default function User({ params, data }: PageProps<typeof loader>) {
  // params: { id: string } / data: User ← どちらも loader 型から推論
  return (
    <div>
      {params.id}: {data.name}
    </div>
  );
}
```

### plugin のライフサイクル

- `configResolved` で初回生成 (build 時もここで走る)
- `configureServer` で `src/routes/` 配下の add / unlink / rename を watch、
  変更時に再生成
- vite の `buildStart` でも走らせる (watch 外の再起動対応)

## Rationale

### 論点 1: 1-B (物理生成)

- plugin の主目的は IDE 体験 (補完 + タイポ検知) の向上。virtual module は
  vite dev を起動してないとタイプチェックが通らない状況を生み、build / tsc
  単体での CI 運用と噛み合わない
- 生成ファイルを `.vidro/` 配下に固定すれば git から除外しやすく、ファイル 1 枚
  のコストは小さい
- Remix / TanStack Router / astro も同方針 (生成ファイル + watch)

### 論点 2: 2-B (module augmentation)

- 生成ファイルが runtime 依存しない declaration のみになる
- ユーザー側は `import { LoaderArgs } from "@vidro/router"` だけ書けば
  `LoaderArgs<"/users/:id">` が補完される (生成ファイルを明示 import する必要なし)
- `RouteMap` が `interface` なので augment 可能 (`type` だと交差で書き換え不可)
- TS の `vite/client`, `csstype` 等と同じパターンで、TypeScript 利用者には
  見慣れた構造

### 論点 3: 3-B (props 完結)

- Vidro は invoke-once で React のような「render ごとに再実行される hook」の
  モデルを持たない。`useX` 接頭辞を持つ API が 1 つも無い現状で hook を 1 個だけ
  入れると文化的に浮く
- hook で型安全を達成するには plugin codegen でファイルごとに型を埋める魔法が
  必要で、Vidro の透明性哲学とはトレードオフ (設計書 5 哲学の #1)
- 深い子孫 component で param が欲しいケースは:
  - 親 component から props で渡す (明示的)
  - どうしても context 的なアクセスが欲しいなら **後から signal を足せる**
- 「leaf は hook、深い子孫は props」のような分岐を作らず、**全部 props で統一**
  することで学習コストを最小化

### 論点 4: 4-A (LoaderArgs helper 拡張)

- 既存の `LoaderArgs<Params>` helper が存在し、ユーザーは既に慣れている
  (Phase 3 第 1 弾 / 0009 でも使ってる)
- generic を `Params` (object 型) から `R extends keyof Routes` に差し替えるだけ
  で、書き心地はそのまま route path の文字列リテラルに切り替わる
- 将来 loader 引数に `request: Request` や worker `context` が追加される想定
  (設計書 3.7 / 3.3)、helper 1 箇所の変更で全 route が追従する構造を維持
- `defineLoader` (4-C) は runtime wrapper が必要で、`export async function loader`
  の素朴さを失う。Vidro は「コンパイラの仕事は JSX の 2 つだけ」と宣言してる
  (設計書 5 哲学の #1) ので、loader は runtime helper 不要の素朴な関数で書かせる
- 4-B (辞書直接) は params が 2 回登場するのが冗長、helper を 1 枚被せる方が
  ユーザー体験が良い

## Consequences

### 実装

- `packages/plugin/src/route-types.ts` 新設:
  - `routeTypes(options?: { root?: string, routesDir?: string, outFile?: string })`
    をデフォルト引数付きで export
  - `configResolved` / `buildStart` / `configureServer` の 3 フックで生成トリガー
  - `chokidar` は vite が既に使ってるので `this.environment.config.server.fs`
    経由ではなく、vite の `watcher.add(routesDir)` + event で十分
  - 出力は `@vidro/router` への `declare module` 1 つのみ (runtime 成果物ゼロ)
- `packages/plugin/src/index.ts` から `routeTypes` を追加 export
- `packages/router/src/page-props.ts`:
  - 新 `interface RouteMap {}` を export (augment target)
  - `type Routes = RouteMap` を export (`keyof Routes` で使う)
  - `LoaderArgs` を `LoaderArgs<R extends keyof Routes = keyof Routes>` に拡張
  - `LoaderArgs` default generic は `keyof Routes` にしておくと、RouteMap 空の
    状態で `LoaderArgs` (bare) 書いたときに `params: Routes[keyof Routes]["params"]`
    → `never` の params になる。初回生成前は `LoaderArgs<R>` を annotate した
    loader がエラーを出す可能性があるが、plugin が最初に走れば解消
- `packages/router/src/index.ts` から `RouteMap` / `Routes` を追加 export

### apps 側

- `apps/router-demo/vite.config.ts` / `apps/website/vite.config.ts` に
  `routeTypes()` を追加
- `apps/*/tsconfig.json` の `include` に `src/.vidro/**/*.d.ts` を含める
  (既に `src` 全体を include しているなら不要)
- `.gitignore` に `src/.vidro/` を追加
- 既存 loader (`LoaderArgs<{ id: string }>` 等) を `LoaderArgs<"/users/:id">`
  形式に移行

### 制約・既知の課題

- 初回 clone 時、vite を一度も起動していない状態では `src/.vidro/routes.d.ts`
  が存在しないため `LoaderArgs<"/users/:id">` が型エラーになる。`vp dev` 起動か
  `vp build` で plugin が走れば解消。CI でも型チェック前に build / dev 相当の
  plugin ステップを走らせる必要がある (実質 `vp check` の前に `vp build` を挟む)
- ファイル rename 直後、旧 key を参照してる loader が一瞬型エラーになる
  (watcher 再生成を待つ間)
- `useParams` hook は今回スコープ外。深い子孫で params が欲しい需要が具体化
  したら別 ADR で signal / context で追加する

### 設計書への影響

- 3.6「Route params: directory から自動生成された型」を実装済みに格上げ
- 3.6 の `useParams<R>()` は Vidro では **採用しない** (props 完結方針) こと
  を反映。代わりに `LoaderArgs<R>` + `PageProps<typeof loader>` の 2 点で
  params 型を通すスタイルを記載

## Revisit when

- **useParams を入れる必要が出てきた時** (深い子孫 component で params を
  context 的に取りたい要求):
  - 案 1: signal で `currentParams: Signal<Record<string, string>>` を
    Router が流し込む (型は緩い)
  - 案 2: plugin codegen でファイルごとに route path を埋め込む (TanStack 流、
    魔法度高)
  - 選択基準は「その時点で hook 形式をユーザーがどれくらい求めてるか」
- **loader 引数に `request` / `context` を追加する時**:
  `LoaderArgs<R>` の中に追加、全 route が helper 経由で追従。request が
  generic にぶら下がらない前提なら引数順序だけ注意
- **route path を型で手書きする機会が増え、文字列 typo が問題化した時**:
  `Routes["/users/:id"]` のドット記法で強制的に key 補完が効くので、その時点で
  既にタイポは防げている。helper 無しで書く 4-B スタイルへの回帰を検討する余地
- **server boundary (案 B) を入れた時**:
  server.ts / layout.server.ts が client bundle から除外される。そのとき
  route 型生成は client-only plugin として切り分けるか、server plugin と
  同居させるかを再検討
